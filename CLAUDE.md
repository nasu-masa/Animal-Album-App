# Project Rules

- 現在は最小構成のMVPを機能単位で実装する
- 依頼されていない将来機能を先回りして実装しない
- 変更前に既存コード、README、設計書、設定ファイルを確認する
- 推測だけでコードや設定を変更しない
- 依頼された範囲だけ変更し、関係のないファイルを触らない
- 既存の命名規則とディレクトリ構成に合わせる
- 不要なファイル、抽象化、パッケージを追加しない
- 重複は避けるが、一度しか使わない短い処理を無理に共通化しない
- 可読性を下げる過剰な責務分割や共通化をしない
- 現在のバージョンで推奨される書き方を使用する
- 非推奨、廃止予定、古いバージョン向けの書き方を使用しない
- DB構造、API仕様、認証方式、Docker構成を勝手に変更しない
- 追加パッケージが必要な場合は、追加前に理由を説明する
- `.env`、秘密情報、依存関係、ビルド生成物をコミットしない
- 変更後は可能な範囲で構文、型、Lint、テスト、動作を確認する
- 実行できなかった確認は、理由とともに明記する
- 作業後は変更ファイル、変更理由、確認結果を簡潔に説明する
