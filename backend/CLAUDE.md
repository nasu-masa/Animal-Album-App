# Backend Rules

- LaravelとPHPの現在のプロジェクトバージョンに従う
- Controllerはリクエストを受け取り、処理を呼び出し、レスポンスを返す役割にとどめる
- Controllerへ長いバリデーションや複雑な処理を詰め込まない
- 入力値の検証は原則としてFormRequestへ分ける
- APIレスポンスは原則としてResourceで整える
- Modelにはリレーション、casts、fillable、Model自身の小さな処理を書く
- 単純な一覧取得や詳細取得だけでServiceを作らない
- 複数処理、ファイル保存、複数Model更新などで必要な場合にServiceを検討する
- 将来利用を理由にRepositoryやInterfaceを作らない
- 同じバリデーション、クエリ、認可、ファイル保存処理を重複させない
- 一度しか使わない短い処理を無理に共通化しない
- DB構造はMigrationとER図を確認する
- MigrationとER図に差異がある場合は勝手に直さず報告する
- 外部キー、unique、nullable、削除時の挙動を明確にする
- TypeScript側で型制限していてもLaravel側で必ず検証する
- リクエスト値を信用せず、許可値、型、文字数、ファイル形式を検証する
- N+1を避け、必要な関連は適切にEager Loadingする
- ループ内で同じクエリを繰り返さない
- APIレスポンスへModelを無条件にそのまま返さない
- ファイル保存にはLaravelのStorageを使用する
- 元のファイル名をそのまま保存パスに使用しない
- ファイル保存とDB登録の失敗による不整合を考慮する
- 認証と認可を分けて考える
- 同じ認可処理が複数箇所にある場合はPolicyを検討する
- 意味のない`try-catch`を追加せず、例外を握りつぶさない
- 不要な`dd()`、`dump()`、デバッグログを残さない
- 変更後は構文、Migration、対象API、テストを可能な範囲で確認する
